# Розбіжності v2 ↔ старий контракт (що враховувати аналітику)

Колонковий контракт (заголовки) кожної з 21 таблиці збережено 1:1 зі старим репо
(`config/schemas/`). Але деякі поля у джерелі v2 відсутні або влаштовані інакше, тож
відповідні колонки заповнюються порожнім (`None`) або мають адаптовану логіку. Нижче — повний
перелік, щоб порожні колонки не сприймались як баг пайплайна.

## Колонки без джерела у v2 (завжди порожні)
- **1**: `number_of_employees_volunteers` — 3-го показника зайнятості у v2 немає.
- **3.1 property_objects**: `object_owner_type` — у realty v2 є `owner_name/owner_code`, але немає типу.
- **3.2 movable_property**: `object_owner` — у movable v2 немає власника (є `manufacturer_name/trade_mark`).
- **3.3 vehicles**: `object_owner`, `object_owner_edrpou`, `object_owner_type`, `object_owner_birth_date` —
  transport v2 має лише `owning_subject_id` (без імені/коду власника) → `check_edrpou` тут no-op.
- **6 in_kind_donations**: `donation_metodological_price` — у v2 один `payment_amount` (→ `donation_contract_price`);
  методологічної ціни немає.
- **9.1 / 9.2 expenditures**: `bank_name`, `bank_EDRPOU`, `account_number`, `account_type` (+ `account_type2` у 9.1) —
  у v2 для витрат заповнено лише отримувача (`receiver_*`); власний рахунок, З ЯКОГО платили, не фіксується по-рядково.

## Поля-здогадки (підтвердити на ручній звірці)
- **6, 8**: `object_registration_number` ← `payment_number` (семантика не документована).
- **8**: `income_description` ← `payment_description`.
- **9.3 / 9.5** (помилкові надходження/повернення): «повертальні» колонки (`returning_*`, `recepient_*`)
  мапляться з `refund_*` та `receiver_*`; структура складна, потребує вибіркової звірки з вебом.

## Адаптовані фільтри порожніх рядків
Старі фільтри спирались на поля, які у v2 стабільно null, тож їх послаблено:
- **5 private_contributions**: старий `donor_name & bank_edrpou notna` → `donor_name.notna()`
  (у v2 `receiver_bank_code` майже завжди null; банк ідентифікується назвою+IBAN).
- **9.2**: старий `bank_EDRPOU & account_number notna` → `recipient_name.notna()`
  (у v2 банк платника не заповнений).

## Дані, відсутні у v2 повністю
- **3.4 securities** (`property_paper`) — **порожня** в усіх 74k звітах → файл лише зі схемою колонок.
- **9.4 false_donations_returning** — окремої таблиці **немає**. У старому репо це був окремий файл
  «повернення внесків та помилкових надходжень»; у v2 повернення відображені всередині таблиць
  9.3 та 9.5 (`returning_*`), окремого джерела під 9.4 немає → файл не формується.

## Таблиці, що можуть бути відсутні (умовні)
- **0_report_duplcates** — формується лише коли в даних є задубльовані звіти (ключ: період, рік,
  партія). v2 позначає видалені/непідписані звіти (`deleted`/`status`/`signed`), тож фантомних
  дублів зазвичай немає й файл не створюється. Це очікувано, а не помилка пайплайна.

## Формати
- `report_submition_date` = `signed_date` v2 (`YYYY-MM-DD HH:MM:SS`), а не старий ISO з `Z`/мс.
- `legal_entity_region` бере значення довідника як є (`"Київ"` / `"м.Київ"` / `"Донецька"` / `"Донецька обл."`
  трапляються обидва) — v2 не нормалізує область; окремо не вирівнюємо.
- Деперсоналізація v2 = літерал `[конфіденційна інформація]` → `None` (старе `***`).
